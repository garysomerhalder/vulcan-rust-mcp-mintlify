## Getting Started Examples

Perfect for learning Vulcan fundamentals:

### Counter Server

<Card title="Counter Server" icon="hashtag" href="/examples-docs/counter-server">
  **Difficulty:** Beginner ⭐
  **Concepts:** Basic tool creation, state management, stdio transport

  A simple stateful server that demonstrates:
  - Creating tools with `#[tool]` macro
  - Managing server state with Arc<Mutex<T>>
  - Stdio transport for Claude Desktop
  - Zero-boilerplate tool registration

  **Perfect for:** First-time Vulcan users, understanding tool basics
</Card>

**Key takeaways:**
```rust
#[tool(description = "Increment counter")]
async fn increment(&self) -> Result<i32, ErrorData> {
    let mut count = self.count.lock().await;
    *count += 1;
    Ok(*count)
}
```

  **Concepts:** Hexagonal architecture, dependency injection, testability

  Production-ready architecture showing:
  - Domain layer with business logic

  - Swappable adapters (InMemory → Postgres)

```

Ports (ToolRepository trait)
```

## Data Integration Examples

Working with databases, APIs, and external services:

### Database Integration

<Card title="Database Integration" icon="database" href="/examples-docs/database-integration">
  **Difficulty:** Intermediate ⭐⭐
  **Concepts:** Async database access, connection pooling, transactions

  Production database patterns:
  - SQLx for type-safe queries
  - Connection pool management
  - Transaction handling
  - Error mapping to ErrorData
  - Migration strategies

  **Perfect for:** Data-driven servers, CRUD operations
</Card>

**Database patterns:**
```rust
#[tool(description = "Create user")]
async fn create_user(
    &self,
    name: String,
    email: String
) -> Result<User, ErrorData> {
    let user = sqlx::query_as!(
        User,
        "INSERT INTO users (name, email) VALUES ($1, $2) RETURNING *",
        name,
        email
    )
    .fetch_one(&self.pool)
    .await?;  // Auto-converts sqlx::Error → ErrorData

    Ok(user)
}
```

## Example Templates

Use these templates to create your own examples:

### Basic Tool Server Template

```rust
use vulcan::prelude::*;

#[derive(Clone)]
struct MyServer {
    tool_router: ToolRouter<Self>,
}

#[tool_router]
impl MyServer {
    fn new() -> Self {
        Self {
            tool_router: Self::tool_router(),
        }
    }

    #[tool(description = "Your tool description")]
    async fn my_tool(&self, param: String) -> Result<String, ErrorData> {
        // Your logic here
        Ok(format!("Processed: {}", param))
    }
}

#[tool_handler]
impl ServerHandler for MyServer {
    fn get_info(&self) -> ServerInfo {
        ServerInfo {
            protocol_version: ProtocolVersion::V_2024_11_05,
            capabilities: ServerCapabilities::builder()
                .enable_tools()
                .build(),
            server_info: Implementation {
                name: "my-server".to_string(),
                version: "1.0.0".to_string(),
                ..Default::default()
            },
            instructions: Some("Server instructions".to_string()),
        }
    }
}

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    let transport = StdioTransport::new();
    MyServer::new().serve(transport).await?;
    Ok(())
}
```

### Middleware-Enabled Server Template

```rust
use vulcan::prelude::*;
use vulcan::adapters::middleware::{LoggingMiddleware, MetricsMiddleware};

#[tokio::main]
async fn main() -> Result<(), Box<dyn std::error::Error>> {
    // Create middleware chain
    let mut chain = MiddlewareChain::new();
    chain.add(Arc::new(LoggingMiddleware::new()));
    chain.add(Arc::new(MetricsMiddleware::new()));

    // Serve with middleware
    let transport = StdioTransport::new();
    MyServer::new().serve_with_middleware(transport, chain).await?;
    Ok(())
}
```

### Database-Backed Server Template

```rust
use sqlx::PgPool;

#[derive(Clone)]
struct DatabaseServer {
    tool_router: ToolRouter<Self>,
    pool: PgPool,
}

impl DatabaseServer {
    async fn new(database_url: &str) -> Result<Self, sqlx::Error> {
        let pool = PgPool::connect(database_url).await?;
        Ok(Self {
            tool_router: Self::tool_router(),
            pool,
        })
    }
}

#[tool_router]
impl DatabaseServer {
    #[tool(description = "Query database")]
    async fn query_data(&self, id: i32) -> Result<Data, ErrorData> {
        let data = sqlx::query_as!(Data, "SELECT * FROM data WHERE id = $1", id)
            .fetch_one(&self.pool)
            .await?;  // sqlx::Error auto-converts to ErrorData
        Ok(data)
    }
}
```

## Contributing Examples

We welcome community examples! Here's how to contribute:

### Example Requirements

<Steps>
  <Step title="1. Choose a Clear Use Case">
    Focus on a specific real-world problem:
    - ✅ "OAuth 2.0 integration with GitHub"
    - ✅ "Streaming responses with SSE"
    - ❌ "Random example" (too vague)
  </Step>

  <Step title="2. Follow Production Patterns">
    Use best practices:
    - Error handling with `ErrorData`
    - Async/await properly
    - Connection pooling for databases
    - Middleware for cross-cutting concerns
    - Comprehensive logging
  </Step>

  <Step title="3. Include Complete Code">
    Provide runnable examples:
    - Full source code
    - Cargo.toml with dependencies
    - README with setup instructions
    - Test suite demonstrating usage
  </Step>

  <Step title="4. Document Thoroughly">
    Explain the "why" not just the "what":
    - Problem statement
    - Architecture decisions
    - Trade-offs made
    - Common pitfalls
    - Production considerations
  </Step>
</Steps>

### Example Structure

```
examples/my-example/
├── README.md           # Overview and setup
├── Cargo.toml          # Dependencies
├── src/
│   ├── main.rs        # Server implementation
│   └── lib.rs         # Reusable components
├── tests/
│   └── integration_test.rs  # Working tests
└── docs/
    └── architecture.md # Design decisions
```

  - [ ] Usage examples
  - [ ] Architecture explanation

  - [ ] Best practices
